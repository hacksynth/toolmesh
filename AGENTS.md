📦 仓库状态 (Repository status)

以事实为准：将仓库中已有的文件视为唯一的“事实来源” (Source of Truth)，绝不擅自假设。

📥 任务接收 (Task intake)
明确任务要素：对于复杂的任务，在进行重大代码编辑之前，请先将请求重述为：目标 (Goal)、背景 (Context)、约束条件 (Constraints) 以及完成标准 (Done when)。

谋定而后动：如果技术栈仍未定义，在选择框架、运行时、包管理器、数据库、托管目标或 CI 系统之前，必须先询问用户确认。

提前计划：在执行耗时较长或包含多个步骤的任务之前先做计划。

🛠️ 代码实现 (Implementation)
小步快跑：优先采用小巧、易于审查的补丁 (patches)，避免进行大规模的全局重写。

控制范围：严格将修改范围限制在请求之内。避免顺手进行无关的代码重构 (drive-by refactors)。

尊重现状：保留用户的更改和不相关的本地编辑。未经明确批准，绝不回滚或覆盖不相关的工作。

最小化新增：尽可能复用现有的脚本、工具和目录结构。如果不存在，则只添加维持运行所需的最少代码/依赖。

并行工作：如果需要并行处理任务，请按文件或职责保持所有权的独立性，避免冲突。

基建文件审查：在执行 git commit 之前，检查开源基础设施文件是否需要更新以反映此次更改，特别是 README.md、LICENSE、CONTRIBUTING.md、CODE_OF_CONDUCT.md、SECURITY.md、.gitignore 以及 GitHub 模板文件。

及时提交：一旦子任务完成并通过了相关验证，请立即创建一个 git commit。

📝 提交规范 (Commit Guidelines)
约定式提交：严格遵循 约定式提交 (Conventional Commits) 规范。

提交格式：提交信息必须采用 <类型>[可选的作用域]: <描述> 的结构（例如：feat(auth): 添加 JWT 登录支持）。

常用类型 (Types)：

feat: 新增功能

fix: 修复 Bug

docs: 仅文档更改

style: 不影响代码含义的更改（空白、格式化、缺少分号等）

refactor: 既不修复 Bug 也不添加功能的代码更改（重构）

perf: 提高性能的代码更改

test: 添加缺失的测试或纠正现有测试

chore: 构建过程或辅助工具和库的更改

要求：确保提交信息清晰、准确，并在涉及破坏性变更 (Breaking Changes) 时在页脚或类型后使用 ! 进行显著标明。

🔗 依赖项与生成文件 (Dependencies and generated files)
询问优先：在添加生产依赖项、锁文件 (lockfiles)、代码生成器、CI/CD 配置、容器化配置或大型生成产物之前，必须先询问用户。

忽略构建产物：不要提交代码生成的输出文件，除非仓库中已经对其进行了版本控制跟踪，或者用户明确要求您提交。

✅ 验证与测试 (Verification)
最小化验证：在每次修改后，运行能覆盖该修改的最小范围的相关验证。

使用现有命令：优先使用仓库中实际配置的代码检查 (lint)、测试 (test)、类型检查 (typecheck) 和构建 (build) 命令。

手动验证反馈：如果目前还没有可用的验证命令，请在回复中明确说明这一点，并详细描述您手动检查了哪些内容。

📖 文档 (Documentation)
同步更新：在引入新的技术栈或新建顶级目录时，务必在同一次 Commit 中更新 README.md 以反映这些架构变化。

记录命令：一旦项目确定了工具链，请在文档中清晰地记录项目的安装 (setup)、运行 (run)、测试 (test) 和发布 (release) 命令。

💬 沟通协作 (Communication)
语言匹配：匹配用户使用的语言。当用户使用中文沟通时，默认使用简洁、专业的中文进行回复。

风险前置：尽早且主动地向用户指出当前的假设前提 (assumptions)、阻碍因素 (blockers) 以及潜在的风险权衡 (risk tradeoffs)。
